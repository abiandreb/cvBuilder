# CV Builder

## Огляд

CV Builder — це повноцінний веб-застосунок, який дозволяє користувачам створювати, редагувати та керувати професійними резюме через покроковий майстер. Користувач реєструє акаунт, заповнює особисті дані, досвід роботи, освіту та навички на шести кроках майстра, обирає один із п'яти візуальних шаблонів і зберігає резюме в базі даних PostgreSQL. З панелі керування він може перейменовувати резюме, змінювати шаблони, експортувати в PDF, замовляти AI-переклад на українську, німецьку або польську мову, а також видаляти резюме. Сесія зберігається через JWT у localStorage, тому після оновлення сторінки користувач залишається авторизованим.

## Як запустити

### Необхідні умови

- Node.js версії 18 або новіше
- Docker Desktop (для PostgreSQL)

### Крок 1 — Запустити PostgreSQL

```bash
docker run -d --name cvbuilder-pg \
  -e POSTGRES_PASSWORD=password \
  -e POSTGRES_DB=cvbuilder \
  -p 5432:5432 \
  postgres:16
```

### Крок 2 — Налаштувати бекенд

```bash
cd cv-api
npm install
npx prisma migrate dev --name init
npm run dev
```

→ API працює на http://localhost:3001

### Крок 3 — Запустити фронтенд

```bash
cd react
npm install
npm run dev
```

→ Застосунок працює на http://localhost:5173

### Змінні середовища

**`cv-api/.env`**

| Змінна | За замовчуванням | Опис |
|--------|-----------------|------|
| `DATABASE_URL` | `postgresql://postgres:password@localhost:5432/cvbuilder` | Рядок підключення PostgreSQL, який використовує `@prisma/adapter-pg` |
| `JWT_SECRET` | *(обов'язково, без значення за замовчуванням)* | Секретний ключ для підпису та перевірки HS256 JWT; задайте будь-який довгий випадковий рядок |

## Схема бази даних

### User

| Поле | Тип | Обмеження | Опис |
|------|-----|-----------|------|
| `id` | `String` | PK, `@default(cuid())` | Колізійно-стійкий унікальний ідентифікатор |
| `name` | `String` | обов'язкове | Ім'я для відображення |
| `email` | `String` | `@unique` | Ідентифікатор для входу |
| `password` | `String` | обов'язкове | Хеш bcrypt (коефіцієнт 10) |
| `createdAt` | `DateTime` | `@default(now())` | Час створення акаунту |
| `cvs` | `Cv[]` | зв'язок | Усі резюме, що належать цьому користувачу |

### Cv

| Поле | Тип | Обмеження | Опис |
|------|-----|-----------|------|
| `id` | `String` | PK, `@default(cuid())` | Колізійно-стійкий унікальний ідентифікатор |
| `userId` | `String` | FK → User.id | Посилання на власника |
| `name` | `String` | обов'язкове | Людинозрозуміла назва резюме |
| `template` | `String` | `@default("classic")` | Ключ шаблону: classic / modern / minimal / executive / vibrant |
| `data` | `Json` | обов'язкове | Весь вміст резюме: personal, summary, experience, education, skills |
| `language` | `String` | `@default("en")` | Код мови ISO 639-1 |
| `isTranslation` | `Boolean` | `@default(false)` | True для резюме, створених функцією перекладу |
| `originalCvId` | `String?` | nullable | ID вихідного резюме, з якого виконано переклад |
| `createdAt` | `DateTime` | `@default(now())` | Час створення запису |
| `updatedAt` | `DateTime` | `@updatedAt` | Автоматично оновлюється при кожному записі |

Зв'язок `Cv.user` має `onDelete: Cascade`, тому видалення користувача автоматично видаляє всі його резюме.

## Нотатки з безпеки

### Що готове до продакшену

- Паролі хешуються bcrypt (коефіцієнт 10) — ніколи не зберігаються у відкритому вигляді.
- JWT підписуються алгоритмом HS256 і закінчуються через 24 години.
- Кожен захищений ендпоінт перевіряє токен перед зверненням до бази даних.
- На кожному CV-ендпоінті перевіряється власність (403 Forbidden, якщо `cv.userId !== userId`).
- Заголовки CORS встановлюються за значенням `Origin` запиту, а не як wildcard, що дозволить коректно передавати cookie після їх впровадження.

### Що є лише для демо і потребує змін у продакшені

| Проблема | Ризик | Вирішення |
|---------|-------|-----------|
| JWT у `localStorage` | Вразливий до XSS — будь-який впроваджений скрипт може прочитати токен | Перенести в `httpOnly`, `Secure`, `SameSite=Strict` cookie |
| Відсутність refresh-токенів | Токен не можна відкликати; вкрадений токен діє повні 24 години | Додати ротацію refresh-токенів + таблицю відкликання |
| Відсутність rate limiting | Брутфорс на `/api/auth/login` тривіальний | Додати `express-rate-limit` або rate limiter у Next.js middleware |
| CORS origin відображає origin запиту | Будь-який сайт може звертатися до API | Обмежити конкретним allowlist дозволених origin |
| Немає обмежень довжини вводу | Надмірно великі JSON-навантаження можуть перевантажити БД | Валідувати й обмежувати довжину полів на стороні сервера |
| Base64-фото у JSON `data` | Великі рядки base64 роздувають рядок у БД | Зберігати фото в object storage (S3, R2); тримати тільки URL |

## Архітектурні рішення (питання та відповіді)

**1. Чому Next.js для бекенду, де тільки API, а не Express?**

Route Handlers Next.js (файли `route.ts` всередині `app/api/`) надають файловий REST API без жодного шаблонного налаштування. Кожен обробник — це звичайна асинхронна функція, що отримує `NextRequest` і повертає `NextResponse`, ідентичні Fetch API — тобто той самий код працює на Node.js, Vercel Edge і Cloudflare Workers без змін. З Express довелось би окремо налаштовувати body-parser, CORS, обробку помилок та бандлер. Next.js також постачається з підтримкою TypeScript з коробки, що важливо для типів, згенерованих Prisma. Недолік — більший розмір залежностей (React у пакеті, хоча він не використовується), але для проєкту такого масштабу переваги для розробника переважають витрати.

**2. Чому в `schema.prisma` немає поля `url` (зміна в Prisma v7)?**

Починаючи з Prisma v7, коли використовується driver adapter, підключення до бази даних повністю керується адаптером (у цьому проєкті — `@prisma/adapter-pg`), а не вбудованим пулом з'єднань Prisma engine. Оскільки адаптер отримує рядок підключення безпосередньо через `new PrismaPg({ connectionString: process.env.DATABASE_URL })`, поле `datasource db { url = ... }` у `schema.prisma` більше не потрібне — Prisma читає його з адаптера під час виконання. Відсутність цього поля також усуває плутанину: рядок підключення вказується лише в одному місці (`lib/prisma.ts`). Це навмисна несумісна зміна в Prisma v7, яка робить підхід через driver adapter канонічним.

**3. Що таке driver adapter у Prisma v7 і навіщо він потрібен?**

У старіших версіях Prisma вбудований Rust-рушій управляв підключеннями до бази даних через libpq (для PostgreSQL). Driver adapter замінює цей Rust-шар JavaScript/TypeScript-драйвером бази даних — у цьому проєкті npm-пакетом `pg`. `@prisma/adapter-pg` реалізує інтерфейс адаптера, транслюючи внутрішній SQL Prisma у виклики `pg.Pool`. Це означає, що весь стек Prisma тепер є чистим JavaScript, що потрібно для роботи в середовищах без підтримки нативних бінарних файлів (Vercel Edge Functions, Cloudflare Workers, Deno). Він також дозволяє керувати стратегіями пулингу з'єднань (PgBouncer, Neon serverless) на рівні драйвера, а не всередині Prisma.

**4. Чому `jose`, а не `jsonwebtoken`?**

Бібліотека `jsonwebtoken` є синхронною і покладається на модуль `crypto` з Node.js — вона не працює в Edge Runtime чи Web Crypto середовищах. `jose` побудована виключно на Web Crypto API (`SubtleCrypto`), тобто вона повністю асинхронна і працює в будь-якому JavaScript-середовищі: Node.js, Deno, Cloudflare Workers, браузер і Vercel Edge Functions. Оскільки цей проєкт використовує Next.js middleware (який за замовчуванням працює на Edge), `jose` є правильним вибором. API також більш явний: `new SignJWT(payload).setProtectedHeader(...).setExpirationTime(...).sign(secret)` чітко розділяє кожен крок, що полегшує аудит.

**5. Чому `bcryptjs`, а не нативний `bcrypt`?**

Нативний пакет `bcrypt` є Node.js-аддоном, який необхідно компілювати з C++-коду під час `npm install`. Ця компіляція потребує `node-gyp`, Python і платформо-специфічних інструментів збірки, що часто ламається на Windows або в CI-середовищах без цих інструментів. `bcryptjs` — чиста JavaScript-реалізація того самого алгоритму bcrypt: без компіляції, без нативних залежностей, ідентичний результат. Властивості безпеки однакові; єдиний компроміс — швидкість, яка тут не важлива, оскільки вартість bcrypt навмисно висока (коефіцієнт 10 означає ~100 мс на хеш на звичайному залізі).

**6. Чому PrismaClient є синглтоном? Що станеться без цього?**

`lib/prisma.ts` зберігає клієнт в об'єкті `global` і повторно використовує його при наступних імпортах. Без цього патерну гаряча заміна модулів Next.js у режимі розробки перевиконувала б модуль при кожній зміні файлу, створюючи новий екземпляр `PrismaClient` — і, відповідно, новий пул з'єднань — кожного разу. PostgreSQL має максимальне обмеження `max_connections` (зазвичай 100 за замовчуванням). Після десятків перезавантажень сервер вичерпав би доступні з'єднання і почав кидати помилки "too many clients". Синглтон гарантує, що протягом усього часу роботи процесу існує лише один пул. У продакшені це менш критично, оскільки модулі завантажуються один раз, але патерн все одно є правильною практикою.

**7. Чому JWT зберігається в localStorage, а не в httpOnly cookie? Які компроміси?**

Зберігання в `localStorage` простіше реалізувати для SPA: `localStorage.setItem('cv_token', token)`, і зчитування при кожному запиті не потребує конфігурації cookie на сервері. Недолік полягає в тому, що `localStorage` доступний будь-якому JavaScript на сторінці, роблячи його вразливим до XSS-атак — успішний XSS-експлойт може викрасти токен. `httpOnly` cookie взагалі не читається JavaScript-ом, що повністю усуває цей вектор атаки. Для продакшену правильним підходом є `httpOnly`, `Secure`, `SameSite=Strict` cookie в поєднанні з CSRF-захистом. Поточна реалізація прийнятна для демо/портфоліо-проєкту, де XSS не є реалістичною загрозою.

**8. Як працює CORS у цьому застосунку? Поясніть preflight-запит OPTIONS.**

Браузер дотримується Same-Origin Policy: сторінці на `localhost:5173` не дозволено читати відповіді від `localhost:3001`, якщо сервер явно це не дозволить. Для "не простих" запитів (з заголовком `Authorization` або JSON-тілом) браузер спочатку надсилає preflight-запит `OPTIONS` на той самий ендпоінт із запитанням: "чи готові ви прийняти цей метод і ці заголовки з цього origin?" `middleware.ts` обробляє це, повертаючи відповідь `204 No Content` із заголовками `Access-Control-Allow-Origin`, `Access-Control-Allow-Methods` і `Access-Control-Allow-Headers`, видаючи дозвіл. Middleware запускається на всіх шляхах, що відповідають `/api/:path*`. Для запитів не `OPTIONS` він просто додає заголовок `Access-Control-Allow-Origin` до відповіді обробника маршруту, дозволяючи браузеру передати відповідь викличному JavaScript.

**9. Чому `CvContext` очищає `cv_list` та `cv_users` з localStorage при монтуванні?**

До додавання бекенду попередня версія застосунку зберігала резюме та акаунти користувачів безпосередньо в `localStorage` — і паролі зберігалися там у відкритому вигляді. `useEffect`, що викликає `localStorage.removeItem('cv_list')` і `localStorage.removeItem('cv_users')` при монтуванні, є одноразовою міграцією даних: він тихо видаляє старі небезпечні дані з будь-якого браузера, де вони ще є, щоб застарілі або небезпечні дані не зберігалися. Це shim зворотньої сумісності, який можна видалити після того, як усі користувачі оновляться.

**10. Чому фронтенд викликає `api.me()` при монтуванні, замість того щоб просто прочитати токен локально?**

JWT можна декодувати на клієнті (це просто Base64-закодований JSON), але декодування — не те саме, що верифікація. Локально декодований токен повідомляє, що стверджує токен, але не те, чи він досі дійсний на сервері, чи існує ще акаунт користувача, і чи не було змінено секрет. Викликаючи `GET /api/auth/me` при монтуванні, `AuthContext` виконує повну серверну верифікацію: `verifyJwt` перевіряє підпис і термін дії, а `prisma.user.findUnique` підтверджує наявність запису користувача. Якщо будь-яка перевірка не проходить, токен видаляється з `localStorage` і користувача перенаправляють на сторінку авторизації. Це запобігає сценарію, коли токен видаленого користувача надавав би доступ до застарілого дашборду.

**11. Як переклад працює від початку до кінця? Як підключити Gemini?**

Коли користувач натискає Translate у `TranslateModal`, викликається `translateCv(cv.data, selectedLang)` з `utils/translate.js`, яка делегує виклик `api.translate(cvData, targetLang)` — HTTP `POST` на `/api/translate`. Обробник маршруту в `app/api/translate/route.ts` наразі симулює 2-секундну затримку AI через `setTimeout` і повертає захардкоджений переклад лише поля `summary`. Щоб підключити Google Gemini, треба замінити цей блок на `const genAI = new GoogleGenerativeAI(process.env.GEMINI_API_KEY)`, сформувати промпт з інструкцією перекласти всі текстові поля на цільову мову, зберігаючи структуру JSON, передати `cvData` як вхідні дані, розпарсити JSON-відповідь і повернути її. Решта пайплайну (створення нового запису резюме з `isTranslation: true` і цільовою мовою) вже працює коректно і не потребувала б змін.

**12. Що таке `cuid()` і чому використовується для ID замість автоінкрементних цілих чисел?**

`cuid()` (Collision-resistant Unique IDentifier) генерує короткі, URL-безпечні ID, наприклад `clx7f3k2g0000abc123defghi`. На відміну від автоінкрементних цілих чисел, CUID генеруються на стороні клієнта (або на рівні ORM) без звернення до бази даних, що робить їх безпечними у розподілених системах із кількома записувачами. Вони не розкривають загальну кількість записів у базі даних (цілочисельний ID `42` дає зрозуміти, що є приблизно 42 користувачі). Їх також безпечно включати безпосередньо в URL без появи передбачуваних перелічуваних значень. Компроміс полягає в тому, що CUID довші за цілі числа і трохи повільніше індексуються, але для масштабу цього застосунку це абсолютно незначно.

**13. Що робить `onDelete: Cascade` у схемі Prisma?**

Директива `onDelete: Cascade` у зв'язку `Cv.user` вказує PostgreSQL автоматично видаляти всі рядки `Cv`, що посилаються на рядок `User`, коли цей рядок `User` видаляється. Без неї спроба видалити користувача, у якого є резюме, викликала б помилку порушення зовнішнього ключа. При каскадному видаленні операція є атомарною: PostgreSQL видаляє користувача і всі пов'язані резюме в одній транзакції. Це правильна поведінка для цієї моделі даних, оскільки резюме без власника позбавлене сенсу. У схемі Prisma це записується як `user User @relation(fields: [userId], references: [id], onDelete: Cascade)`.

**14. Як реалізація CORS у `middleware.ts` обробляє preflight-запити?**

`middleware.ts` зареєстрований із конфігурацією `matcher: '/api/:path*'`, тому він запускається перед кожним обробником маршруту в `/api/`. Коли вхідний метод — `OPTIONS`, функція повертає одразу `204`-відповідь із трьома CORS-заголовками (`Access-Control-Allow-Origin`, `Access-Control-Allow-Methods`, `Access-Control-Allow-Headers`) — обробник маршруту взагалі не виконується. Це важливо: якби `OPTIONS`-запит дійшов до обробника маршруту, Next.js повернув би `405 Method Not Allowed`, оскільки маршрут експортує лише `GET`/`POST`/`PATCH`/`DELETE`. Для всіх інших методів `NextResponse.next()` передає керування обробнику маршруту, а middleware додає заголовки `Access-Control-Allow-Origin` і `Access-Control-Allow-Headers` до відповіді перед тим, як вона досягає браузера.

**15. Якщо відкрито дві вкладки браузера і користувач видаляє резюме в одній, що станеться в іншій?**

Нічого автоматично. Список резюме у `CvContext` — це стан React у пам'яті. Коли вкладка А видаляє резюме, вона викликає `api.deleteCv(id)` і оновлює локальний стан через `setCvs(prev => prev.filter(...))`. Вкладка Б не має механізму отримання цього оновлення — в застосунку немає WebSocket, Server-Sent Events або polling. Масив `cvs` у вкладці Б досі містить видалену картку. Якщо користувач у вкладці Б спробує відредагувати, перейменувати або перекласти цю картку, API поверне `404 Not Found`, і помилка з'явиться в консолі (або в catch-обробнику). Оновлення вкладки Б вирішує неузгодженість. Для продакшену варто реалізувати polling `GET /api/cvs` з коротким інтервалом, використати `BroadcastChannel` для синхронізації стану між вкладками одного origin, або впровадити WebSocket push-сповіщення.

**16. Як додати rate limiting до API?**

Найпростіший підхід для Next.js API — додати логіку rate limiting всередині `middleware.ts`, який вже перехоплює кожен запит до `/api/*`. Потрібно підтримувати `Map<string, { count: number, resetAt: number }>` з ключем за IP клієнта (заголовок `req.ip` або `x-forwarded-for`), збільшувати лічильник при кожному запиті і повертати `429 Too Many Requests`, якщо лічильник перевищує поріг у межах часового вікна. Для продакшен-деплою пам'ять процесу не підходить, оскільки кілька екземплярів сервера не розділяють стан — треба використовувати Redis (наприклад, через `ioredis` і алгоритм sliding window) або керований сервіс, як Upstash Rate Limit. Встановлення суворіших лімітів саме для `/api/auth/login` і `/api/auth/register` є найпріоритетнішою зміною для захисту від brute-force атак.

**17. Який SQL генерує `prisma.cv.findMany({ where: { userId }, orderBy: { updatedAt: 'desc' } })`?**

Prisma транслює цей виклик у наступний параметризований SQL-запит:

```sql
SELECT
  "id", "userId", "name", "template", "data",
  "language", "isTranslation", "originalCvId",
  "createdAt", "updatedAt"
FROM "Cv"
WHERE "userId" = $1
ORDER BY "updatedAt" DESC;
```

Плейсхолдер `$1` зв'язується з фактичним рядком `userId` під час виконання драйвером `pg`, що запобігає SQL-ін'єкції. Prisma не використовує `SELECT *` — він перераховує всі стовпці, визначені у схемі. `ORDER BY "updatedAt" DESC` безпосередньо відповідає `orderBy: { updatedAt: 'desc' }`. Щоб зробити цей запит ефективним при великій кількості даних, до схеми слід додати складений індекс на `("userId", "updatedAt" DESC)` за допомогою `@@index([userId, updatedAt(sort: Desc)])`.

## Сценарій демонстрації

Виконайте ці кроки для показу повного набору функцій по черзі:

1. Відкрийте http://localhost:5173 — завантажується лендинг із живим попереднім переглядом трьох шаблонів.
2. Натисніть **Start Building** — оскільки ви не авторизовані, вас перенаправить на сторінку Auth.
3. Перейдіть на вкладку **Register**, заповніть ім'я, email та пароль (мінімум 6 символів), натисніть **Create account** — потрапляєте на дашборд із написом "No CVs yet".
4. Натисніть **New CV** — відкривається майстер на кроці 1 (Personal). Заповніть ім'я та email (обов'язкові поля), спостерігайте як живий перегляд праворуч оновлюється в реальному часі.
5. Натискайте **Continue** через кроки Summary, Experience, Education, Skills, заповнюючи хоча б часткові дані на кожному кроці.
6. На кроці 6 (Preview) оберіть інший шаблон із селектора — повнорозмірний перегляд перемикається миттєво. Натисніть **Save & Exit**.
7. На дашборді з'явилась картка вашого резюме з бейджем обраного шаблону та сьогоднішньою датою.
8. **Двічі клікніть** на назву картки, щоб перейменувати її на місці — натисніть Enter або клікніть в інше місце для збереження.
9. Натисніть **Design** на картці — з'являється дропдаун із п'ятьма шаблонами. Оберіть інший — мініатюра картки одразу перерендериться і зміни збережуться.
10. Натисніть **Translate** — відкривається модальне вікно. Оберіть **Ukrainian** і натисніть **Translate** — з'являється спінер на 2 секунди, потім вікно закривається і на дашборді з'являється нова картка з бейджем прапора.
11. На будь-якій картці натисніть **Download PDF** — кнопка показує "Generating…" поки html2canvas захоплює перегляд, потім у папку завантажень зберігається PDF-файл.
12. Натисніть **Delete** на картці — з'являється діалог підтвердження; підтвердіть — картка зникає зі сітки.
13. Оновіть сторінку (F5) — застосунок викликає `GET /api/auth/me` зі збереженим токеном, сесія відновлюється, дашборд завантажується з рештою ваших резюме.
14. Натисніть **Log out** — токен видаляється з localStorage і вас повертає на лендинг.

## Відомі обмеження та плани на майбутнє

- **Переклад Gemini не підключений** — ендпоінт `/api/translate` використовує захардкоджений mock, що замінює лише поле `summary`. Змінна середовища `GEMINI_API_KEY` згадується в коментарях, але ніколи не читається.
- **Відсутність refresh-токенів** — JWT дійсний 24 години без можливості його відкликати або тихо видати новий.
- **Відсутність пагінації** — `GET /api/cvs` повертає весь список резюме користувача. Для великої кількості резюме це стає повільним.
- **Base64-фото в БД** — поле `data.personal.photo` приймає рядок фото в кодуванні Base64, що може займати сотні кілобайт на одне резюме. Для продакшену фото слід завантажувати в object storage (AWS S3, Cloudflare R2) і зберігати в БД лише URL.
- **Відсутність підтвердження email** — користувачі можуть реєструватися з будь-яким email без підтвердження власності.
- **Відсутність тестів** — немає ні юніт-, ні інтеграційних тестів ні для API-маршрутів, ні для React-компонентів.
- **Вміст може розділятися посередині елемента** — багатосторінковий PDF розрізає канвас на фіксованих межах A4, тому довгий абзац або запис досвіду може бути розбитий між сторінками. Повноцінне вирішення потребує рендерингу в нативні PDF-примітиви (наприклад, `@react-pdf/renderer`) замість знімку канвасу.
